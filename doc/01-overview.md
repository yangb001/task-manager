# 桌面任务管理 App — 架构方案

## 一、技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 桌面框架 | **Electron** | Node.js 生态，原生支持系统托盘后台常驻 |
| 前端 | **React + TypeScript** | 组件化，生态成熟 |
| UI 库 | **Ant Design** | 表单/表格/弹窗开箱即用 |
| 状态管理 | **Zustand** | 轻量，无 boilerplate |
| 本地数据库 | **SQLite** (sql.js WASM) | 零配置，嵌入式，WASM 跨平台兼容 |
| 任务调度 | **node-cron** + 自建时间轮 | Cron 表达式 + 精确单次调度 |
| 邮件收发 | **imapflow** (收) + **nodemailer** (发) | 纯 JS，IMAP/SMTP |
| 敏感存储 | **Electron safeStorage** | 系统级加密存密码/Token |
| 构建 | **electron-builder** | 打包分发 |

### 进程职责划分

**主进程（Main Process）** — 所有"干活"的部分
- 任务调度引擎（定时器、Cron）
- 触发器引擎（轮询 Welink/邮件/文件）
- 动作执行（发 Welink 消息、发邮件、执行命令）
- 数据库读写
- 系统托盘管理

**渲染进程（Renderer Process）** — 纯 UI
- 任务列表展示、新建/编辑表单
- 执行历史查看
- 状态实时更新（通过 IPC 推送）

**为什么这样分？**
- 用户关窗口时，渲染进程可以关掉，但主进程继续在托盘运行
- 任务调度和轮询不受 UI 影响
- 安全性：渲染进程不直接操作数据库
